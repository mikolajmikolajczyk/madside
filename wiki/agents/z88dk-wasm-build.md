# Rebuilding `z80asm.wasm` + `appmake.wasm` (z88dk)

The z88dk assembler/linker (`z80asm`) and tape packager (`appmake`) ship as
wasm32-wasip1 binaries for the ZX toolchain (asm-first; the C path — sccz80/zsdcc
— is deferred, issue #87). Same class as the cc65 wasm build: wasi-sdk clang,
`@bjorn3/browser_wasi_shim` at runtime. Pins in `third-party.toml`; owned build
inputs in `build-support/z88dk/`; scratch in `_notes/z88dk-wasm-spike/build/`
(git-ignored).

## One command

```sh
just build-z88dk-wasm
```

1. Fetches wasi-sdk (shared with the cc65 build).
2. Clones z88dk + the `ext/{regex,optparse,uthash}` submodules at pinned commits.
3. **Patches** the checkout (see *Mods* below) — all build-time, no fork.
4. Compiles `z80asm` (C++17) and `appmake` (C) to wasm.
5. Installs them next to the toolchain plugin loader; runs the smoke test.

## Host requirements

`PATH`: `git`, `perl`, `node` (≥18, WASI smoke), wasi-sdk clang (auto-fetched).
**RAM: ~8 GB** with the parse1 split applied (see below); the split is mandatory —
without it the `parse1.c` compile needs **~28 GB** and will OOM a normal machine.

## Why this is non-trivial (the gate spike)

Unlike mads/cc65 (ready-made wasm targets), z88dk had to be extracted from its
autotools build and ported to WASI. Findings worth keeping:

- **z80asm is C++17** (not plain C as first assumed): `src/*.cpp`, `src/cpp/*.cpp`
  + `src/c/*.c` + `../common/*.c` + `ext/regex/reg*.c`. Uses `std::filesystem`
  (libc++ provides it) and `execinfo` stack traces (auto-off when not glibc, so a
  no-op under WASI).
- **Generated parsers are committed** (`parse_rules.h` 8 MB, `scan_rules.h`,
  `cpu_rules*.h`, `scan2.cpp`) — no re2c/ragel/perl needed at compile time,
  *except* when regenerating (see the split step).
- **`config.h`** is 4 trivial `#define`s — generated by the recipe, not the
  autotools machinery.
- **appmake** is plain C; only `ti8xk.c` (TI calculator) needs `-lgmp` — excluded.

### The 28 GB → 8 GB fix (parse1.c)

`parse_rules.h` is one ragel-generated function `_parse_statement_1` (~46 k lines,
a 15 k-case action switch). The wasm backend pass **`WebAssemblyFixIrreducibleControlFlow`**
(`processRegion`) is ~O(n²) in basic blocks and OOMs at ~28 GB on this single
huge irreducible loop body — wasm-specific (native gcc/clang build it fine; this
is why z88dk CI on small runners works).

Levers that did **not** help: `-fno-jump-tables`, IR-level `fix-irreducible`,
`lower-switch`, `-O0/-O2`, ragel `-F1` (produced a 6.6 GB file) / `-G` (worse).
The frontend + IR-gen are cheap (<12 GB); only machine codegen explodes.

Fix: **`build-support/z88dk/split-action-switch.py`** extracts the action switch
out of the driver loop into a loop-free helper `_parser_exec_action()`. The driver
loop shrinks to a few hundred blocks (cheap); the helper has no loops (the pass
does nothing). `fgoto` actions (`goto _again`) become `return 1` + the caller
re-loops. Result: parse1.c compiles in <8 GB, byte-identical output. This is what
makes the build CI-able (and therefore npm-publishable later).

## Mods (all patch-at-build, no fork)

Mirrors the cc65 pattern — pin upstream, patch the checkout in the recipe:

- `build-support/z88dk/split-action-switch.py` — the parse1.c split above
  (regex-anchored, robust against minor regen drift).
- `build-support/z88dk/wasm-stubs.c` — `system()` (WASI has no fork/exec; z80asm
  uses it only for the m4 path + an unused modlink branch — both off the asm
  flow) and `mkstemp()` (absent from wasi-libc; appmake's chain temp files need
  it; implemented over WASI `open()`).
- `appmake.h` guard: extend z88dk's own `#ifndef _MSC_VER` ti8xk stub (their
  PR #2433 disables GMP for msbuild) to also honour `-DNO_GMP`. One-line `perl`
  patch in the recipe.
- Compile flags: `-fwasm-exceptions -lunwind` (std::filesystem throws),
  `-Wno-error=implicit-function-declaration` (mkstemp).

## Runtime notes (for the toolchain plugin, #84)

- The WASI VFS must contain `/tmp` — appmake hardcodes `/tmp/appmakechainXXXXXXXX`.
- appmake `-o NAME` writes exactly `NAME` (no `.tap` suffix) — pass the full name.
- Orchestrate in JS (no `zcc` driver — WASI can't fork): `z80asm -b -mz80` →
  `.bin`, then `appmake +zx --binfile … --org … -o out.tap`.

## Smoke test

`build-support/z88dk/smoke.asm` → `z80asm -b` → 5 bytes `3e 02 d3 fe c9`
(`ld a,2 / out (0xfe),a / ret`) → `appmake +zx` → a valid `.tap` (BASIC loader
`CLEAR 32767 : LOAD ""CODE : RANDOMIZE USR 32768` + CODE block).

## Outputs

`z80asm.wasm` ≈ 12 MB, `appmake.wasm` ≈ 700 KB.

## When to rebuild

Don't rebuild casually. Bump the pins in `third-party.toml`, rerun
`just build-z88dk-wasm`, smoke-test, then commit the new wasm. If a pin bump
changes `parse_rules.h`'s structure, re-check `split-action-switch.py`'s anchors.
