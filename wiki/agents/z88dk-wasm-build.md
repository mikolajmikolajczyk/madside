# Rebuilding `z80asm.wasm` + `appmake.wasm` (z88dk)

The z88dk assembler/linker (`z80asm`) and tape packager (`appmake`) ship as
wasm32-wasip1 binaries for the ZX toolchain, plus the **C path** (`zcc` driver +
`ucpp`/`zpragma`/`sccz80`) ‚Äî see "C path (#87)" below; `just build-z88dk-c`. Same
class as the cc65 wasm build: wasi-sdk clang,
`@bjorn3/browser_wasi_shim` at runtime. Pins in `third-party.toml`; owned build
inputs in `build-support/z88dk/`; scratch in `_notes/z88dk-wasm-spike/build/`
(git-ignored).

## One command

```sh
just build-z88dk-wasm
```

1. Fetches wasi-sdk (shared with the cc65 build).
2. Clones z88dk + the `ext/{regex,optparse,uthash}` submodules at pinned commits.
3. **Patches** the checkout (see *Mods* below) ‚Äî all build-time, no fork.
4. Compiles `z80asm` (C++17) and `appmake` (C) to wasm.
5. Installs them next to the toolchain plugin loader; runs the smoke test.

## Host requirements

`PATH`: `git`, `perl`, `node` (‚â•18, WASI smoke), wasi-sdk clang (auto-fetched).
**RAM: ~8 GB** with the parse1 split applied (see below); the split is mandatory ‚Äî
without it the `parse1.c` compile needs **~28 GB** and will OOM a normal machine.

## Why this is non-trivial (the gate spike)

Unlike mads/cc65 (ready-made wasm targets), z88dk had to be extracted from its
autotools build and ported to WASI. Findings worth keeping:

- **z80asm is C++17** (not plain C as first assumed): `src/*.cpp`, `src/cpp/*.cpp`
  + `src/c/*.c` + `../common/*.c` + `ext/regex/reg*.c`. Uses `std::filesystem`
  (libc++ provides it) and `execinfo` stack traces (auto-off when not glibc, so a
  no-op under WASI).
- **Generated parsers are committed** (`parse_rules.h` 8 MB, `scan_rules.h`,
  `cpu_rules*.h`, `scan2.cpp`) ‚Äî no re2c/ragel/perl needed at compile time,
  *except* when regenerating (see the split step).
- **`config.h`** is 4 trivial `#define`s ‚Äî generated by the recipe, not the
  autotools machinery.
- **appmake** is plain C; only `ti8xk.c` (TI calculator) needs `-lgmp` ‚Äî excluded.

### The 28 GB ‚Üí 8 GB fix (parse1.c)

`parse_rules.h` is one ragel-generated function `_parse_statement_1` (~46 k lines,
a 15 k-case action switch). The wasm backend pass **`WebAssemblyFixIrreducibleControlFlow`**
(`processRegion`) is ~O(n¬≤) in basic blocks and OOMs at ~28 GB on this single
huge irreducible loop body ‚Äî wasm-specific (native gcc/clang build it fine; this
is why z88dk CI on small runners works).

Levers that did **not** help: `-fno-jump-tables`, IR-level `fix-irreducible`,
`lower-switch`, `-O0/-O2`, ragel `-F1` (produced a 6.6 GB file) / `-G` (worse).
The frontend + IR-gen are cheap (<12 GB); only machine codegen explodes.

Fix: **`build-support/z88dk/split-action-switch.py`** extracts the action switch
out of the driver loop into a loop-free helper `_parser_exec_action()`. The driver
loop shrinks to a few hundred blocks (cheap); the helper has no loops (the pass
does nothing). `fgoto` actions (`goto _again`) become `return 1` + the caller
re-loops. Result: parse1.c compiles in <8 GB, byte-identical output. This is what
makes the build CI-able (and therefore npm-publishable later).

## Mods (all patch-at-build, no fork)

Mirrors the cc65 pattern ‚Äî pin upstream, patch the checkout in the recipe:

- `build-support/z88dk/split-action-switch.py` ‚Äî the parse1.c split above
  (regex-anchored, robust against minor regen drift).
- `build-support/z88dk/wasm-stubs.c` ‚Äî `system()` (WASI has no fork/exec; z80asm
  uses it only for the m4 path + an unused modlink branch ‚Äî both off the asm
  flow) and `mkstemp()` (absent from wasi-libc; appmake's chain temp files need
  it; implemented over WASI `open()`).
- `appmake.h` guard: extend z88dk's own `#ifndef _MSC_VER` ti8xk stub (their
  PR #2433 disables GMP for msbuild) to also honour `-DNO_GMP`. One-line `perl`
  patch in the recipe.
- Compile flags: `-fwasm-exceptions -lunwind` (std::filesystem throws),
  `-Wno-error=implicit-function-declaration` (mkstemp).

## Runtime notes (for the toolchain plugin, #84)

- The WASI VFS must contain `/tmp` ‚Äî appmake hardcodes `/tmp/appmakechainXXXXXXXX`.
- appmake `-o NAME` writes exactly `NAME` (no `.tap` suffix) ‚Äî pass the full name.
- Orchestrate in JS (no `zcc` driver ‚Äî WASI can't fork): `z80asm -b -mz80` ‚Üí
  `.bin`, then `appmake +zx --binfile ‚Ä¶ --org ‚Ä¶ -o out.tap`.

## Smoke test

`build-support/z88dk/smoke.asm` ‚Üí `z80asm -b` ‚Üí 5 bytes `3e 02 d3 fe c9`
(`ld a,2 / out (0xfe),a / ret`) ‚Üí `appmake +zx` ‚Üí a valid `.tap` (BASIC loader
`CLEAR 32767 : LOAD ""CODE : RANDOMIZE USR 32768` + CODE block).

## Outputs

`z80asm.wasm` ‚âà 12 MB, `appmake.wasm` ‚âà 700 KB.

## When to rebuild

Don't rebuild casually. Bump the pins in `third-party.toml`, rerun
`just build-z88dk-wasm`, smoke-test, then commit the new wasm. If a pin bump
changes `parse_rules.h`'s structure, re-check `split-action-switch.py`'s anchors.

## C path (#87) ‚Äî work-in-progress notes

The asm path ships; C is #87. The tooling is **de-risked + all three compilers
build to WASI** (spikes live in `_notes/z88dk-wasm-spike/`, git-ignored):

- **`sccz80.wasm`** (650K) ‚Äî z88dk's native C compiler. Plain gnu99, builds clean
  with wasi-sdk clang. Needed one stub: `tmpfile()` (wasi-libc lacks it) over
  mkstemp+fdopen, in `build-support/z88dk/wasm-stubs.c`. Recipe: `build-sccz80.sh`.
  Proven: compiles `.c` ‚Üí z88dk `.asm` (SECTION code_compiler + C_LINE + l_* runtime calls).
- **`zcpp.wasm`** (379K) ‚Äî z88dk's ucpp C preprocessor (`src/ucpp`, `-DSTAND_ALONE
  -DUCPP_CONFIG`). **REQUIRED** before sccz80: sccz80 does NOT expand `#define`
  (it emits the macro name as a symbol) or handle `#include`. ucpp uses
  setjmp/longjmp ‚Üí must build with **`-mllvm -wasm-enable-sjlj`** + link
  `-fwasm-exceptions -lunwind -lsetjmp`. Runs under node WASI (engine has wasm EH);
  proven: `N*2` with `#define N 5` ‚Üí `5*2`. Recipe: `build-ucpp.sh`.
- **`z80asm.wasm`** (the committed asm one) reads the prebuilt v2.4 libs:
  our HEAD z80asm writes object format **`Z80RMF18`**, v2.4's `.lib` are
  **`Z80LMF18`** ‚Äî same v18, **compatible, no re-pin of the z80asm source needed**.

**Prebuilt +zx C library is available** (the git tree's `lib/clibs` is empty ‚Äî
artifacts; the **release** has them). From `z88dk-osx-2.4.zip` (GitHub release
`v2.4`, libs are platform-independent):
- `lib/clibs/zx_clib.lib` (1.2M, classic clib) + `lib/clibs/mzx.lib` (47K, ZX maths)
- classic crt0 `lib/target/zx/classic/spec_crt0.asm` (17.9K) + `lib/config/zx.cfg`
- headers `include/` + `include/arch/zx/`
- (the "new" `_DEVELOPMENT` clib at `libsrc/_DEVELOPMENT/lib/sccz80/zx.lib`
  exists too, but its link is zcc-orchestrated/harder ‚Äî prefer the **classic** path.)

### Chosen approach: ship `zcc.wasm` + shim `system()` (NOT recipe-replication)

The driver problem (`zcc` fork/exec's its sub-tools; WASI has none) is solved by
**shimming `system()` to a host import** rather than reverse-engineering zcc's
classic link. **Proven end-to-end** (spike `/tmp/shimtest`, but the technique is
the keeper):

- Override `system(cmd)` in zcc.wasm to call an imported host function
  (`__attribute__((import_module("env"),import_name("run")))`). Confirmed: it
  compiles with wasi-sdk clang and shows up as a wasm import `env.run`.
- The JS host provides `env.run(cmdPtr)`: read the C string from the instance's
  memory, parse the shell-ish command (`z88dk-sccz80 ‚Ä¶` / `z88dk-z80asm ‚Ä¶` /
  `zcpp ‚Ä¶` / `z88dk-appmake ‚Ä¶`), dispatch to that sub-tool's wasm module, run it
  **synchronously** on the **shared VFS**, return its exit code.
- **Proven loop:** a `system()`-shimmed test wasm called `env.run("z88dk-sccz80
  -o /x.asm /x.c")` ‚Üí the host ran the *real* `sccz80.wasm` on the shared preopen
  ‚Üí it produced `x.asm` (49 lines) ‚Üí returned 0 ‚Üí the caller saw rc=0.

#### Built + wired so far (spikes in `_notes/z88dk-wasm-spike/`)

`zcc.wasm` (452K) **runs as the driver** and emits the full classic `+zx` recipe
through the shim. Build it from `src/zcc`: sources `zcc.c` + `../copt/regex/reg*.c`
+ `../common/{dirname,option}.c`, flags `-DLOCAL_REGEXP -I. -I../copt -I../common
-I../../ext/uthash/src` (NOT ext/regex ‚Äî zcc uses its own copt/regex), plus
`-Wno-error=implicit-function-declaration -Wno-error=int-conversion`. The shim
(`build/zcc-shim.c`) overrides `system()`‚Üí`env.run` and adds `mkstemp`/`mkdtemp`/
`tmpfile` (wasi-libc lacks them). Recipe: `build-zcc.sh`.

**The captured classic recipe** (from `zcc +zx -v -create-app`), i.e. what the
dispatcher must service:
```
z88dk-ucpp  hello.c -o X.i2                       # preprocess   ‚Üí zcpp.wasm
z88dk-zpragma -sccz80 -zcc-opt=zcc_opt.def <X.i2 >X.i   # pragma‚Üíopt  ‚Üí (passthrough ok for no-pragma)
z88dk-sccz80  X.i -o X.opt                          # compile      ‚Üí sccz80.wasm
z88dk-copt -mz80 z80rules.9 <X.opt >X.op1          # peephole √ó3  ‚Üí (passthrough = unoptimised but correct)
z88dk-copt -mz80 z80rules.2 <X.op1 >X.opt
z88dk-copt -mz80 z80rules.1 <X.opt >X.asm
cat X.asm >> X.tmp ; cat X.tmp >> X.asm            # (zcc merge dance)
z88dk-z80asm -s -mz80 -I.../lib X.asm              # assemble     ‚Üí z80asm.wasm
cat spec_crt0.asm >> Y.asm ; ‚Ä¶                     # classic crt0
z88dk-z80asm -s -mz80 -I.../lib -I.../zx/classic Y.asm
<LINK: objects + libs>                             # final link   ‚Üí z80asm.wasm -b -d ‚Ä¶
```
- `z88dk-copt` and `z88dk-zpragma` are **also C** (`src/copt/copt.c`,
  `src/zpragma/zpragma.c`) ‚Äî buildable, but copt's `regex/` collides with
  wasi-libc's `<regex.h>`; for a first cut the **dispatcher treats both as
  passthrough** (copy stdin‚Üístdout): the program compiles correct but unoptimised,
  and a no-`#pragma` hello needs no zcc_opt.def beyond zcc's own defaults. Build
  them properly later for optimisation + pragma support.
- The host dispatcher (node spike `_notes/.../zxbuild-dispatch.mjs`) tokenises the
  command (quotes + `< > >>` redirection, drop literal `(null)` args), services
  `cat`, passthroughs, and runs the 4 real tools via `node:wasi` with file-backed
  stdin/stdout (open the redirect targets, pass fds as `WASI({stdin,stdout})`).
  ZCCCFG=`/z88dk/lib/config`; sysroot = the v2.4 `lib/` + `include/` mounted in
  the preopen (DESTDIR resolves to `ZCCCFG/../..`).

**Two zcc-shim blockers found + FIXED (both are real, keep them):**
1. **`/dev/null`** ‚Äî `zcc_vasprintf()` (zcc.c) measured format length by opening
   `/dev/null` + `vfprintf`; WASI has no `/dev/null` ‚Üí `fopen` returns NULL ‚Üí it
   returned `*s=NULL` ‚Üí **every command zcc built came out empty** (the earlier
   "empty linker" symptom). Fix: patch `zcc_vasprintf` to measure with
   `vsnprintf(NULL,0,‚Ä¶)` (build-time patch; no /dev/null needed). After this, all
   commands carry full args.
2. **Quote stripping** ‚Äî the dispatcher's tokenizer only stripped *fully*-quoted
   tokens, so `-zcc-opt="‚Ä¶/zcc_opt.def"` reached sccz80 with literal quotes ‚Üí
   "Can't open zcc_opt.def". Fix: a proper char-by-char shell tokenizer that drops
   quote chars *within* tokens.

After both fixes the chain runs ucpp‚Üízpragma‚Üísccz80‚Üícopt(passthrough)‚Üíz80asm
(main) + crt0 assemble. **Every tool is verified working in isolation** (ucpp,
zpragma, sccz80, z80asm; the full manual chain on hello.c yields a 293-line `.opt`
with `GLOBAL _main`).

**WASI `..` path traversal ‚Äî FIXED ‚Üí C COMPILES, LINKS, AND BOOTS end-to-end.**
zcc derives DESTDIR = `ZCCCFG/../..`, passing ucpp `-isystem"/z88dk/lib/config/../..//include"`;
WASI's sandboxed `path_open` won't traverse `..`, so ucpp silently expanded no
`#include` ‚Üí `_main`-less object ‚Üí link `undefined _main`. **Fix: the host
dispatcher normalises `..`/`.`/`//` in every tool arg's path portion before exec
(`normPathPart`).** After that the full chain runs and **`zcc +zx -create-app`
produces a working binary + `.tap`** (a minimal `int main(){*(char*)0x4000=0xff;}`
builds clean, exit 0). **Verified it BOOTS in the chips zx core: the C poke landed
(`screen[0x4000]=0xff`).** The binary starts `cd 31 80` (= z88dk crt0 `CALL`, org
0x8000). **The C path works.**

Remaining polish (not blockers): **`printf`/stdio** pulls the console driver and
fails `undefined symbol: writebyte` ‚Äî the ZX console-driver lib/pragma isn't wired
yet (a no-stdio C program links fine). And **copt/zpragma**: zpragma is real now;
copt is still passthrough (unoptimised). Then: package the +zx sysroot zip, install
zcc/zcpp/sccz80/zpragma wasm in the recipe, port the dispatcher to browser_wasi_shim
(shared in-memory Directory + the `..` normaliser + the `vsnprintf` zcc patch),
wire toolchain-z88dk (inputExt c/h ‚Üí run zcc), add a zx-c-hello template, pin
z88dk v2.4 in third-party.toml.

**Why this beats replicating zcc's recipe:** zcc stays the driver, so it owns ALL
the classic-link logic (crt0, libs, `zcc_opt.def`, `CRT_*` defines, link order) ‚Äî
**zero reverse-engineering**, works for any z88dk feature, survives version bumps
(just rebuild). Cost: ship `zcc.wasm` + the 4 sub-tools + a host dispatcher +
shared VFS.

Shared VFS: **node WASI uses the real FS** (preopen the same dir ‚Üí sharing is
free, as in the spike). The **browser** path (`browser_wasi_shim`, in-memory FS)
must pass the **same `PreopenDirectory`/`Directory` object** to every instance so
files flow between tools ‚Äî the one runtime-integration piece to verify.
